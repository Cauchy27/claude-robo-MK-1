# Sonnet アップリフトスキル - 詳細リファレンス

このファイルは [SKILL.md](./SKILL.md) の詳細リファレンスです。
必要なセクションだけを読み込んで使用すること（遅延読み込み推奨）。

---

## 1. 背景と狙い

### コスト構造の非対称性

| モデル層 | 役割 | トークン消費 |
|---------|------|-------------|
| 上位モデル（親セッションモデル） | 計画・設計・検証・最終判断 | 全体の1-2割 |
| sonnet / haiku（実行エージェント） | 定型作業の実行 | 全体の8-9割 |

エージェント運用のトークン消費は実行フェーズに集中する。実行だけを低コストモデルに委譲し、
品質の門番（設計・検証）を上位モデルが握れば、品質を維持したままコストを大幅に削減できる。

### フレームワークが埋める差・埋めない差

| 領域 | 効果 | 理由 |
|------|------|------|
| 定型タスク（集計・定型文書・診断・チェックリスト駆動作業） | 高 | プロセスの質はコンテキストで移植できる |
| 形式・一貫性・既知の失敗パターン回避 | 高 | 明文化された基準への照合は下位モデルでも確実に実行できる |
| 未知の難問・長期計画・微妙なトレードオフ判断 | 低 | 素の推論力の差はコンテキストでは埋まらない |

**この非対称性こそがエスカレーション基準を必須構成要素とする理由。**
フレームワークは「下位モデルを上位モデルにする」のではなく、
「下位モデルで足りる仕事と足りない仕事を実行時に仕分ける」仕組みである。

## 2. アップリフト5原則の詳細

### 原則1: 役割分担ルーティング

- **Why:** 品質劣化は「誰が実行するか」より「誰が検収するか」で決まる。検収基準の設計と最終判断を上位モデルに固定すれば、実行モデルを下げても品質の下限が守られる
- **How:** チェックリスト設計（Phase 1）は必ず親セッションモデルが行う。ESCALATION の受け口（SKILL.md 参照）を運用に組み込む

### 原則2: 検証ループはプロンプトに勝る

- **Why:** 下位モデルの単発出力は上位モデルに劣るが、「ドラフト→明文化された基準への照合→修正」のループを回した出力は単発比で大きく改善する。改善の主因は指示の丁寧さではなく検証の機械化
- **How:** 品質ゲートに自己検証ループ（最大2周）を必ず含める。「丁寧にやること」のような訓示を増やしても効果はない

### 原則3: エスカレーションは必須構成要素

- **Why:** セクション1の通り、推論の深さは移植できない。下位モデルに「できない仕事を粘らせる」ことが品質事故の最大要因
- **How:** 既定3条件（セクション4）を必ず含め、粘るより差し戻す方が正しい行動だとエージェント定義に明記する

### 原則4: コンテキスト税の規律

- **Why:** 埋め込みが肥大すると「Prompt is too long」の一因になり、また常駐ルール化すると全ターンにコストが波及する（配置ポリシー違反）
- **How:** ブロックは30-50行以内。配置はエージェント定義内のみ（発動時にだけ読まれる）。CLAUDE.md / .claude/rules/ に汎用ルールとして追加しない

### 原則5: 計測なき改善は願望

- **Why:** 「Fable級になった」は計測しなければただの願望。効果の出ないブロックはコンテキスト税だけを支払う負債になる
- **How:** セクション5の効果検証を適用時に1回、その後は不具合発生時に実施。差が出なければ縮小・撤去

## 3. チェックリスト設計原則（上位モデルの仕事）

### 3.1 設計ルール

| # | ルール | 良い例 | 悪い例 |
|---|--------|--------|--------|
| 1 | 二値判定可能な条件で書く | 「全データに出典が明記されている」 | 「データの信頼性が適切」 |
| 2 | 観測可能な対象を指す | 「結論が冒頭3行以内にある」 | 「結論ファーストである」 |
| 3 | 失敗実績から逆算する | 過去に数値転記ミス→「原文と数値が一致する」 | 起きたことのない失敗の網羅 |
| 4 | 5-9項目に絞る | 主要な失敗モードのみ | 20項目の完全網羅（形骸化する） |
| 5 | Who×How 分離を守る | 判定基準はエージェント側、手順はスキル側 | エージェント定義に作業手順を移植 |

### 3.2 設計者の制約

- **設計は必ず親セッションモデル（上位モデル）が行う。** 実行モデル自身に自分の検収基準を書かせると、自分が通りやすい基準になる（自己採点バイアス）
- 起草後、「このチェックリストを全て満たすが業務的には失格の出力」を1つ想像してみる。想像できたら項目が足りないか抽象的すぎる

### 3.3 既知の失敗パターン欄

ブロック末尾に「既知の失敗パターン」を1-3個記載する。運用中に実際の失敗が発生したら、
チェックリスト項目化するか失敗パターン欄に追記する（フレームワークを実績で育てる）。

## 4. エスカレーション設計

### 4.1 既定3条件（全エージェント共通・削除禁止）

1. 自己検証ループ2周でチェックリストが収束しない
2. チェックリストにないトレードオフ判断が必要になった
3. 不可逆操作・対外公開・機密・法務/人事に関わる判断が必要になった

### 4.2 フラグ仕様

```
ESCALATION: {理由を1文}
完了済み: {完了した部分}
残作業: {残っている判断・作業}
判断材料: {親が判断するために必要な情報}
```

- 出力の**冒頭**に置く（親が見落とさないため）
- 部分的な成果物は捨てずに「完了済み」として返す（再実行コストの節約）

### 4.3 運用ルール

| ルール | 内容 |
|--------|------|
| 受け口必須 | 親は ESCALATION を検知したら SKILL.md の受け口テーブルに従い必ず処理する |
| 無視禁止 | ESCALATION 付き出力をそのまま成果物として採用しない |
| 頻度監視 | 特定エージェントで頻発する場合、そのエージェントの降格が過剰（親継承へ戻す）か、チェックリストが実態と乖離している |

## 5. 効果検証手順

### 5.1 A/B 比較（適用時に1回）

1. 対象エージェントの代表タスクを1-2件選ぶ（境界ケースを含める）
2. 適用前定義と適用後定義で同一タスクを Task 実行する（結果は `.tmp/sonnet-uplift/` に書き出す）
3. 親モデルが両出力を比較し、チェックリスト観点での差分を判定する

### 5.2 判定基準

| 結果 | 対応 |
|------|------|
| 適用後が明確に改善 | 採用 |
| 差が出ない | ブロックを縮小（効いていない項目を削る）か撤去 |
| 適用後が悪化（過剰な形式化・萎縮） | チェックリスト項目を減らす。5項目まで削っても悪化するなら撤去 |

### 5.3 形骸化の検出

運用中、以下の兆候があればチェックリストを見直す:

- 全項目が常に OK（項目が抽象的すぎる）
- ESCALATION がゼロ件のまま失敗が発生（基準が緩い）
- ESCALATION が頻発（降格が過剰、または基準が厳しすぎる）

## 6. 適用対象の選び方と降格判定

### 6.1 適用優先順位（sonnet/haiku 指定済みエージェント）

| 優先 | 条件 | 例 |
|------|------|-----|
| 高 | 使用頻度が高い × 失敗時の手戻りが大きい | 定型レポート生成、デプロイ実行系、対外文面（sonnet） |
| 中 | 使用頻度が高い × 失敗が軽微 | ログ調査、一次調査系 |
| 低 | 使用頻度が低い | 稀にしか呼ばれない診断系 |

### 6.2 親継承エージェントの降格判定（sonnet + 品質ゲートへの変更）

`agent-model-tuning.md` の判定フローに、品質ゲートを織り込んだ第2判定を加える:

```
1. agent-model-tuning.md の分類基準で再判定する
2. 「定型に見えるが判断を含む」と判定された場合のみ、第2判定へ:
   Q: その判断は、二値判定可能な完了条件チェックリストに書き下せるか？
   ├── YES → sonnet + 品質ゲート埋め込みの降格候補（提案 → ユーザー承認 → 適用）
   └── NO  → 親継承を維持
3. 以下は品質ゲートの有無に関わらず降格対象外（agent-model-tuning の原則を上書きしない）:
   - 対人機微（人事評価・メンタル・1on1 等）
   - 経営判断・戦略立案・法務/税務判断
   - GO/NO-GO 判定（リリース可否・品質ゲート運用）
   - 根本原因分析（RCA）・アーキテクチャ設計・コードレビュー
```

**降格は必ず「提案 → ユーザー承認」の2段階。勝手に model を書き換えない。**

### 6.3 haiku への適用注意

haiku は自己検証ループ自体の実行精度も下がる。チェックリストは5項目以内・条件は単純照合のみに絞る。
判断を1つでも含むなら haiku ではなく sonnet にする。

### 6.4 スキル内サブエージェントへの適用

`.claude/skills/*/agents/*.md` にも同じテンプレートを適用できる。
対象は「並列実行で多用される定型サブエージェント」（テスター、フォーマッタ、一次収集係等）。
親スキルのオーケストレーターがエスカレーションの受け口を担う。

## 7. 限界（正直な期待値設定）

| 誤解 | 実際 |
|------|------|
| sonnet が Fable/Opus 級になる | 定型タスクの品質差はほぼ消えるが、推論の深さは変わらない |
| チェックリストを増やせば安全 | 10項目を超えると照合が形骸化し、逆に品質が下がる |
| 一度設計すれば恒久 | 業務・失敗モードは変化する。失敗発生時にチェックリストを更新する運用が前提 |
| 全エージェントに適用すべき | 判断が主務のエージェントに適用しても、判断の質は上がらない（セクション6.2の対象外リスト） |

**このフレームワークの本質は「Sonnet の強化」ではなく「上位モデルを門番に限定する運用設計」である。**

## 10. 品質レベル（QL）と改訂ロードマップ

品質ゲートは「一度書いたら終わり」ではなく、実績で育てる。各エージェントのゲートを以下の3レベルで管理する。

### 10.1 品質レベル定義

| レベル | 状態 | 昇格基準 |
|--------|------|---------|
| **QL1: 初期設計** | 門番が主務・想定失敗モードから演繹で設計。実績データなし | 初期配備時の状態（敵対的検証 PASS が配備条件） |
| **QL2: 実績反映** | 運用で発生した実失敗をチェックリストか既知の失敗パターンに反映済み | 実失敗の反映1件以上、または運用30日での見直し実施 |
| **QL3: 計測済み** | A/B 比較で項目の効き目を計測し、効かない項目を削った状態 | セクション5の効果検証を実施し、結果を記録 |

### 10.2 調整基準（品質が「微妙」と感じた時の判断表）

| 症状 | 診断 | 調整 |
|------|------|------|
| 全項目が常に OK | 項目が抽象的すぎて形骸化 | 二値条件に書き直す（セクション3） |
| ESCALATION がゼロのまま失敗が発生 | 基準が緩い | 既定3条件にタスク固有条件を追加 |
| ESCALATION が頻発（目安: 3割超） | 降格が過剰、または基準が厳しすぎる | 親継承に戻す、または固有条件を削る |
| 出力が形式的・萎縮する | 項目過多 | 5項目まで削減。それでも悪化するなら撤去 |
| 同じ失敗が繰り返される | 実績が未反映 | 失敗をチェックリスト項目化（QL2 昇格） |

### 10.3 改訂ロードマップ

| 版 | 時期 | 内容 |
|----|------|------|
| v1.0 | 2026-07-03（完了） | QL1 一斉配備 — テンプレ39体＋横展開27箇所・計300ゲート |
| v1.1 | 運用開始後随時 | 失敗実績の還流による QL2 化。優先: 並列多用の research / infrastructure 系 |
| v1.2 | 随時 | ESCALATION 発生率の定期観測と受け口運用の定着確認 |
| v2.0 | 判断材料が揃い次第 | 代表タスクの A/B 計測による QL3 化・項目パージ。親継承降格の次弾実施判断（2026-07-03 の初回判定は候補0） |

### 10.4 調整の実務手順

1. 対象エージェントの「## 品質ゲート（sonnet-uplift）」ブロックを直接編集する（テンプレリポが正）
2. 横展開先の同名エージェントへはブロックを転写する（実態が乖離したカスタム機はチェックリストのみ調整）
3. 大きな基準変更は本 reference.md を改訂し、SKILL.md の version を上げて templates/ と同期する

## 11. テスト先行アップリフト（コード実装タスク向けの強化形態）

コード実装タスクでは、チェックリストの代わりに**実行可能なブラックボックステスト**（入力→期待出力）を完了条件にできる。
照合者がモデル自身からテストランナー（決定的・自己採点バイアスなし）に変わるため、品質ゲートの最強形態となる。

### 11.1 品質ゲートとの対応

| 品質ゲートの部品 | テスト先行版での置き換え | 強化される点 |
|-----------------|----------------------|-------------|
| 完了条件チェックリスト | ブラックボックステスト（入力→期待出力） | 条件が実行可能になり解釈の余地が消える |
| 自己検証ループ（モデルが照合） | テストランナー照合（red → 修正 → re-run） | 自己採点バイアスがゼロになる |
| 設計は門番 | テストは仕様から門番が先行生成 | 実装者による検収基準の書き換えを禁止できる |
| エスカレーション既定3条件 | N周で全緑にならない／テストが仕様と矛盾／テスト変更が必要になった | 「テストを直したい」と思った瞬間が差し戻し条件になる |

### 11.2 フロー

```
門番: 仕様確定（入力→期待出力）→ テスト先行生成（凍結）
  → 実行層(sonnet): 実装 → テストランナー照合 → 赤なら修正（N周まで）
  → 全緑のみ成果物へ / 収束しない・テスト矛盾は ESCALATION で門番へ
```

図解: 本テンプレリポの `docs/images/test-first-uplift-loop.svg`（README にも掲載）

### 11.3 絶対ルール

1. **テストは実装者が変更禁止**（凍結）。変更が必要と感じたら ESCALATION する — 仕様変更は門番とユーザーの管轄
2. テスト生成は必ず門番（親セッションモデル）が仕様から先行実施する。実装後にテストを書かない・実装に合わせてテストを直さない
3. 実装コミットへのテスト変更の混入は機械的にブロックする（`templates/hooks/scripts/test-freeze-check.sh`・オプトイン雛形）

### 11.4 塞ぐべき穴と対策

| 穴 | 対策 |
|----|------|
| テスト改竄（テストを直せば緑になる） | test-freeze-check.sh フックでブロック＋コミット diff にテスト変更が無いことを検収で確認 |
| テストへの過適合（期待値ハードコード） | 門番だけが保持する隠しテスト（実装者に見せない追加ケース）を検収時に実行する |
| 全緑＝品質の錯覚（網羅不足） | テスト設計時に「全テスト通過するが失格の実装を想像できるか」を自問する（セクション3.2 と同じ検証） |

### 11.5 限界と併用

テストで測れるのは機能的品質のみ。可読性・設計の筋・保守性は測れないため、
通常の品質ゲート／impl-review-loop（実装→レビュー→修正）と併用するハイブリッドが正しい構成。
置き換えではなく「コード実装タスク限定のゲート強化」と位置付ける。

### 11.6 関連スキルの配線（配布先に存在する場合）

| フェーズ | スキル |
|---------|--------|
| 仕様確定 | spec-planning（デバッグシート先行作成） |
| テスト先行生成（門番） | test-code-generation |
| 実装・修正ループ | testing / impl-review-loop |
