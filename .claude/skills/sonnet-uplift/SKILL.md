# Sonnet アップリフトスキル

## 概要

実行コストの大半を占める定型作業を sonnet/haiku に委譲しつつ、品質の門番を上位モデルに固定するためのフレームワーク適用スキル。
上位モデルが設計した「品質ゲート」（完了条件チェックリスト・自己検証ループ・エスカレーション基準）をエージェント定義に埋め込む。
コンテキストフレームワークが移植できるのは「プロセスの質」であり「推論の深さ」ではないため、エスカレーションを必須構成要素とする。

## 使用タイミング

- model: sonnet / haiku 指定エージェントの出力品質を強化したい時
- 親継承（model 指定なし）エージェントの sonnet 降格を検討する時（降格判定込み）
- agent-creation スキルで新規エージェントに model を指定した直後
- スキル内サブエージェント（`.claude/skills/*/agents/*.md`）の定型実行を強化したい時

## 設計思想（アップリフト5原則）

| # | 原則 | 要点 |
|---|------|------|
| 1 | 役割分担ルーティング | 設計・検証・最終判断は上位モデル（親）、実行は sonnet/haiku。品質の門番を上位モデルに固定する |
| 2 | 検証ループはプロンプトに勝る | 単発出力に期待しない。出力前の自己検証ループ（照合→修正→再照合、最大2周）を埋め込む |
| 3 | エスカレーションは必須構成要素 | 収束しない・基準外のトレードオフ・不可逆/対外/機密は実行モデルに粘らせず親へ差し戻す |
| 4 | コンテキスト税の規律 | 埋め込みは1エージェント30-50行以内。rules/ 等の常駐コンテキストには追加しない |
| 5 | 計測なき改善は願望 | 適用前後を同一タスクで比較する。差が出なければブロックを縮小・撤去する |

詳細: [reference.md](./reference.md) セクション2

## 適用フロー

### Phase 0: 対象選定

1. `grep -rl "^model:" .claude/agents/` で model 指定済みエージェントを列挙する
2. `.claude/docs/agent-model-tuning.md` の判定記録・分類基準を確認する
3. 親継承エージェントの降格を検討する場合は [reference.md](./reference.md) セクション6 の降格判定を実施する
4. **対象一覧（エージェント / 現状 / 適用内容）を提示し、ユーザー承認を得てから進む。勝手に書き換えない**

### Phase 1: 設計（上位モデルの仕事）

1. 対象エージェントの主務・出力・想定読者を1文で書く
2. 失敗モードを列挙する（過去の失敗実績 > 憶測）
3. 完了条件チェックリストを起草する（5-9項目・二値判定可能な条件のみ）
4. エスカレーション基準を確定する（既定3条件＋タスク固有条件）

**設計は必ず親セッションモデルが行う。sonnet 自身に自分のチェックリストを書かせない。**
設計原則の詳細: [reference.md](./reference.md) セクション3

### Phase 2: 埋め込み

1. [templates/uplift-block-template.md](./templates/uplift-block-template.md) を Read する
2. プレースホルダーを Phase 1 の設計内容で置換する
3. 対象エージェント定義の「判断基準」（相当セクション）の直後に「品質ゲート（sonnet-uplift）」セクションを追記する
4. 30-50行以内に収める。frontmatter は変更しない（model の変更は agent-model-tuning のフローで別途実施）

### Phase 3: 検証

1. Task ツールで対象エージェントを起動し、チェックリスト照合と ESCALATION 動作を確認する
2. 可能なら同一タスクを適用前後で実行し差分を比較する（[reference.md](./reference.md) セクション5）
3. エージェント修正後の必須レビューとして internal-structure-review スキルを実施する

### Phase 4: 登録・同期

1. model を変更した場合: `templates/agents.md`（SoT）・本文推奨モデル欄・frontmatter の3点を同期する
2. スキル内サブエージェントに適用した場合: `templates/skills/{name}/agents/` に同期する
3. `scripts/validate-structure.sh` で機械検証する

## エスカレーションの受け口（オーケストレーター側・必須）

Task 結果の冒頭に `ESCALATION:` がある場合、呼び出し側（親モデル）は以下のいずれかを実施する:

| 対応 | 条件 |
|------|------|
| 親が直接引き取る | 残っているのが判断のみの場合 |
| 上位判断エージェント（model 指定なし）に委託 | 専門ペルソナの判断が必要な場合 |
| ユーザーに確認 | 不可逆操作・対外公開・機密に関わる場合 |

**ESCALATION を無視して次工程に進むことは禁止。**

## テスト先行アップリフト（コード実装タスク）

コード実装では、チェックリストの代わりに門番が仕様から先行生成した**ブラックボックステスト**（入力→期待出力）を完了条件にする。
照合者がテストランナーになり自己採点バイアスが消える最強形態。テストは凍結し、実装者の変更を禁止する（変更が必要になったら ESCALATION）。
詳細（対応表・穴と対策・関連スキル配線）: [reference.md](./reference.md) セクション11

- 効果比較などで起動するサブエージェントは結果を `.tmp/sonnet-uplift/` に Write し、メインへの返答は3-5行の要約のみとする

- タスク完了時: `rm -rf .tmp/sonnet-uplift/`

## Troubleshooting

### チェックリストが形骸化する（全項目が常にOK）

**対策:** 項目が抽象的すぎる。二値判定可能な条件に書き直す。→ [reference.md](./reference.md) セクション3

### ESCALATION が一度も発生しない

**対策:** 基準が緩すぎるか、発動テストのタスクが簡単すぎる。境界ケースのタスクで再テストする。→ [reference.md](./reference.md) セクション4

### 適用後も品質が上がらない

**対策:** 対象業務が「推論の深さ」を要する可能性が高い。降格を取り消し親継承に戻す。→ [reference.md](./reference.md) セクション7
